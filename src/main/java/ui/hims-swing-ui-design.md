# HIMS Swing UI 설계 문서

## 목적

현재 HIMS 프로젝트의 재고 및 입출고 관리 시나리오를 발표와 수동 시연에 적합한 Swing UI로 구성한다. 이번 범위에서는 기존 서비스, DAO, DB를 연결하지 않고 UI
전용 더미 데이터로 흐름을 진행한다.

## 전제

- 시작점은 `src/main/java/Main.java`이다.
- UI 코드는 `src/main/java/ui` 패키지 아래에 둔다.
- 서비스 계층은 호출하지 않는다.
- 더미 데이터는 UI 내부 상태로만 관리한다.
- 테스트 코드는 작성하지 않는다.
- 검증은 Gradle 컴파일과 수동 실행으로 한다.

## 핵심 시나리오

발표 흐름은 다음 순서로 진행한다.

1. 로그인 화면에서 사용자 역할을 선택한다.
2. 역할별 대시보드로 이동한다.
3. 입점매장 담당자가 매장 재고를 조회하고 안전재고 이하 상품을 확인한다.
4. 입점매장 담당자가 부족 상품 발주 요청을 생성한다.
5. 발주처 담당자가 발주 요청을 조회하고 승인 또는 반려한다.
6. 발주처 담당자가 승인된 발주를 외부 발주처로 전송하고 출고 완료 상태로 변경한다.
7. 입점매장 담당자가 출고 완료 발주를 입고 확인한다.
8. 정상 입고 시 매장 재고가 증가하고 발주 상태가 완료된다.
9. 판매 처리 시 매장 재고가 차감된다.
10. 지점 관리자는 전체 재고와 마스터 데이터 등록 화면을 확인한다.
11. 시스템 관리자는 직원 계정 관리 화면을 확인한다.

## 역할별 화면

### 로그인 화면

- 로그인 ID 입력 필드
- 비밀번호 입력 필드
- 역할 선택 콤보박스
- 로그인 버튼
- 로그인 성공 시 선택한 역할의 대시보드로 이동
- 로그인 실패 로직은 실제 인증 없이 입력 누락만 확인

### 공통 대시보드

- 상단에 시스템명, 현재 사용자, 현재 역할 표시
- 좌측에 역할별 메뉴 버튼 배치
- 중앙 영역은 선택한 메뉴의 화면으로 교체
- 우측 또는 하단에 현재 시나리오 상태와 처리 메시지 표시
- 로그아웃 버튼으로 로그인 화면 복귀

### 입점매장 담당자 화면

포함 메뉴는 다음과 같다.

- 내 매장 재고 조회
- 안전재고 부족 상품 조회
- 발주 요청 생성
- 입고 검수 대상 조회
- 정상 입고 처리
- 입고 수량 차이 처리
- 입고 반려 처리
- 입고 이력 조회
- 판매 처리

재고 목록은 테이블로 표시한다. 현재수량이 안전재고수량 이하인 행은 별도 색상으로 강조한다.

### 발주처 담당자 화면

포함 메뉴는 다음과 같다.

- 발주 요청 목록 조회
- 발주 요청 상세 확인
- 발주 승인
- 발주 반려
- 승인 발주 외부 전송
- 외부 출고 처리
- 발주 상태별 필터링
- 발주 승인/반려 이력 조회

발주 상태는 `REQUESTED`, `APPROVED`, `REJECTED_BY_VENDOR`, `SENT_TO_VENDOR`, `SHIPPED`,
`RETURNED_BY_STORE`, `COMPLETED`를 사용한다.

### 지점 관리자 화면

포함 메뉴는 다음과 같다.

- 전체 재고 현황 조회
- 지점별 재고 조회
- 브랜드별 재고 조회
- 카테고리별 재고 조회
- 입점매장 정보 조회
- 입점매장 등록 폼
- 상품 등록 폼

등록 폼은 실제 저장 없이 입력값을 화면 로그에 남긴다.

### 시스템 관리자 화면

포함 메뉴는 다음과 같다.

- 직원 계정 목록
- 직원 계정 생성 폼
- 직원 권한 변경 폼
- 직원 계정 정지 처리

계정 생성과 정지는 더미 직원 목록에만 반영한다.

## UI 데이터 모델

UI 전용 모델 클래스는 새로 만들지 않고, 이미 정의된 DTO를 사용한다.

- `employee.EmployeeDTO`: 로그인 사용자, 직원 목록, 직원 계정 생성/정지
- `store.StoreDTO`: 입점매장 조회와 입점매장 등록
- `product.ProductDTO`: 상품 등록과 상품 선택
- `inventory.InventoryDTO`: 매장 재고 조회, 안전재고 부족 조회, 판매 처리
- `order.request.OrderRequestDTO`: 발주 요청, 승인, 반려, 외부 전송, 출고 상태
- `store.receipt.StoreReceiptDTO`: 입고 확인, 입고 차이, 입고 반려, 입고 이력

기존 DTO에 생성자나 보조 접근자가 부족하면 Lombok 어노테이션은 기존 DTO에만 추가한다. 발주/입고 화면에서 DTO에 없는 매장명, 상품명 같은 표시값은 `MockDataStore`의 조회 메서드로 보완한다.

## 상태 전이

발주 요청 상태 전이는 다음과 같이 구현한다.

1. 발주 요청 생성: `REQUESTED`
2. 발주처 승인: `APPROVED`
3. 발주처 반려: `REJECTED_BY_VENDOR`
4. 외부 전송: `SENT_TO_VENDOR`
5. 외부 출고 처리: `SHIPPED`
6. 정상 입고: `COMPLETED`
7. 수량 차이 입고: `RETURNED_BY_STORE`
8. 입고 반려: `RETURNED_BY_STORE`

재고 수량 변경은 다음 시점에만 반영한다.

- 정상 입고: 현재수량이 입고수량만큼 증가
- 판매 처리: 현재수량이 판매수량만큼 감소

## 화면 구성 원칙

- 발표자가 흐름을 따라가기 쉽도록 메뉴명을 기능 명세 ID와 유사하게 둔다.
- 한 화면에서 너무 많은 폼을 보여주지 않고 메뉴 선택에 따라 중앙 패널을 교체한다.
- 테이블과 폼 중심으로 구성해 관리 시스템의 느낌을 유지한다.
- 모든 처리 버튼은 성공 또는 실패 메시지를 화면 로그에 남긴다.
- 실제 서비스 연결 전까지는 예외 상황을 UI 입력 검증 수준으로만 제한한다.

## 제외 범위

- 기존 서비스, DAO, DB 연결
- 외부 발주처 API 호출
- 실제 로그인 인증
- 실제 파일, 엑셀, 이미지 처리
- 테스트 코드 작성
- Oracle 스케줄러, 프로시저 실행
- 복잡한 동시성 처리

## 승인 기준

- `Main.java` 실행 시 Swing 로그인 화면이 표시된다.
- 로그인 후 선택 역할에 맞는 메뉴가 표시된다.
- 더미 데이터 기준으로 핵심 발표 흐름을 끝까지 진행할 수 있다.
- 발주, 승인, 출고, 입고, 판매에 따른 상태와 수량 변경이 화면에 반영된다.
- 서비스 계층 호출 없이 UI 패키지 내부 코드만으로 동작한다.
